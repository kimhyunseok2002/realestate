# 살아남을지도 · SurviMap

> **입지·업종별 상권 생존 예측 서비스** — 위치 + 업종을 넣으면, 그 자리에서 그 업종이 1년 / 3년 / 5년 뒤 살아남을 확률을 예측하고, 무엇이 생존율을 깎고 받쳐주는지 분해해 자연어 리포트로 돌려줍니다.
>
> 브랜드명 **살아남을지도** = "(장사가) 살아남을지도" + "살아남을 지도(地圖)" — 생존 예측과 지도 기반을 한 단어에 담은 중의적 워드마크. 영문 **SurviMap** (Survive + Map). 팀: 김현석 · 최지훈.

기획 문서의 핵심 구조 — **“ML 예측 엔진 + LLM 인터페이스”** — 를 그대로 구현한 작동 프로토타입입니다.

- **예측 그 자체는 ML 코어**(생존분석 Cox·Kaplan-Meier + Weibull)가 담당합니다.
- **LLM(Claude)은 예측 엔진이 아니라** 그 숫자를 소상공인이 읽는 자연어로 **번역·설명하는 레이어**입니다. (판단 대체 ✗ / 검증 지원 ✓)

---

## 빠른 실행

```bash
cd ~/Desktop/상권생존예측
python3 run.py
# 또는  ./run.sh
```

브라우저에서 **http://127.0.0.1:8000** 접속 → 강남역·카페로 자동 예측된 대시보드가 뜹니다.
지도를 클릭하거나 주소를 검색하고, 업종을 골라 **“생존율 예측하기”**.

> 포트 변경: `PORT=8080 python3 run.py`

### 필요 조건
- Python 3.10+ (numpy·scipy·pandas 시스템 설치 가정)
- `fastapi`·`uvicorn` 은 `backend/vendor/` 에 이미 포함되어 별도 설치 불필요
- 인터넷 (지도 타일·주소검색·CDN 라이브러리)
- **Claude CLI** (있으면 실제 LLM 리포트, 없으면 자동으로 템플릿 리포트로 폴백 — 서비스는 항상 동작)

FastAPI (backend/app.py)  ── 오케스트레이션: 데이터 → 예측 → 리포트
  ├ survival.py   ML 예측 엔진 (Cox 비례위험 + Weibull + Kaplan-Meier + 위험분해)
  ├ data.py       서울 자치구 × 업종 피처 데이터셋 (보정된 프로토타입)
  ├ geocode.py    주소 ↔ 좌표 ↔ 자치구 (OpenStreetMap Nominatim)
  └ llm.py        Claude CLI 호출로 리포트/what-if 자연어화 (+ 템플릿 폴백)

### API
| 메서드 | 경로 | 설명 |
| --- | --- | --- |
| GET | `/api/meta` | 자치구·업종 목록, LLM 연결 여부 |
| GET | `/api/geocode?q=` | 주소/상호 → 좌표 후보 |
| GET | `/api/reverse?lat=&lon=` | 좌표 → 주소 + 자치구 |
| POST | `/api/predict` | `{industry,lat,lon,gu?}` → 생존율·곡선·위험분해·유사상권 |
| POST | `/api/report` | `{gu,industry,lat,lon}` → LLM 자연어 리포트 |
| POST | `/api/whatif` | `{...,question}` → 업종/조건 변경 시나리오 재계산 + 설명 |

## ML 예측 엔진 방법론

`backend/survival.py` 는 교과서적 생존분석을 그대로 구현합니다.

1. **Cox 비례위험** — 표준화된 상권 변수 `z` 에 업종별 회귀계수 `β` 를 곱해 선형예측자 `LP = Σβⱼzⱼ`, 위험비 `HR = exp(LP)`.
2. **Weibull 기저 생존함수** — `S₀(t) = exp(-(t/scale)^shape)`. `shape<1` 이 외식업의 초기 폐업위험이 큰 ‘죽음의 계곡’을 표현.
3. **점포 생존함수** — `S(t) = S₀(t)^HR`.
4. **Kaplan-Meier** — 같은 위험모형과 정합한 합성 코호트를 만들어 비모수 생존곡선을 실제로 추정 → “유사 점포들의 실측 생존 이력”.
5. **위험 분해** — 각 변수의 LP 기여분을 3년 생존율 변화(%p)로 환산.

## 프로토타입 데이터 → 실데이터 교체

지금 `backend/data.py` 의 수치는 **서울 상권 경향에 맞춰 손으로 보정한 합성 데이터**입니다.
스키마(필드 이름)는 그대로 두고 값만 아래 공개 데이터로 채우면 엔진은 수정 없이 동작합니다.

| 역할 | 데이터 | 출처 |
| --- | --- | --- |
| 라벨(생존기간) | 인허가 데이터 | 공공데이터포털 / 구 LOCALDATA |
| 점포 변수 | 상가정보(업종·좌표·경쟁밀도) | 공공데이터포털 |
| 상권 변수 | 매출·유동인구·임대료 | 서울 우리마을가게 상권분석 |
| 상권 변수 | 임대료·공실률 | 한국부동산원 R-ONE |
| 배후 변수 | 인구·사업체 일자리 | 통계청 SGIS |

> 문서의 제언대로 **서울 특정 자치구 + 1~2개 업종**부터 실데이터로 검증한 뒤 확장하는 것이 안전합니다.
> 현재는 **수도권(서울 15개 자치구 + 경기 31개 시·군) × 10개 업종**을 지원하며, 미지원 지점은 가장 가까운 지원 상권으로 근사합니다.
> 또한 지역별 **상가 매물**(월세·보증금·관리비·권리금·면적·층)을 생성해, 매물을 클릭하면 그 지점의 생존 예측 + **임대료 대비 예상매출 부담** 분석을 제공합니다. (실 매물 API는 `backend/listings.py`만 교체하면 연결됩니다.)

## LLM(Claude) 설정

- 백엔드가 로컬 **Claude CLI** 를 `claude -p` (비대화 출력) 로 호출합니다. API 키 관리 불필요.
- CLI 위치는 환경변수 `CLAUDE_CODE_EXECPATH` 로 자동 탐지, 없으면 템플릿 리포트로 폴백.
- 모델·타임아웃 조정:
  ```bash
  SANGKWON_LLM_MODEL=claude-haiku-4-5-20251001 SANGKWON_LLM_TIMEOUT=90 python3 run.py
  ```
- 리포트 생성 시에만 호출되므로(예측 루프 밖) API 비용이 리포트 단가 대비 미미 — 문서의 마진 논리와 일치.
